== 什么是 WSL ?

[.my3]
[options="autowidth" cols="1a,1a"]
|===
|Header 1 |Header 2

|什么是 WSL
|WSL (Windows Subsystem for Linux)，即“适用于 Linux 的 Windows 子系统”。**它允许开发者在 Windows 计算机上直接运行 Linux 环境（包括命令行工具、实用程序和应用程序），而无需传统虚拟机（VM）**的沉重开销, 或设置双系统的繁琐。

|WSL的版本
|WSL 经历了两个主要版本的迭代，其底层架构有本质区别：

- WSL 1: 这是一个翻译层。当你运行一个 Linux 命令时，WSL 1 会实时将其转换为 Windows 内核能够理解的系统调用。*它的缺点是对某些复杂的 Linux 系统调用支持不全，且文件 I/O 性能较慢。*
- WSL 2 (推荐): 这是一个真正的轻量级虚拟机。它包含了一个由微软维护的完整 Linux 内核，运行在 Hyper-V 虚拟化技术之上。
|===

|我问:我单独在 LM studio 中运行模型, 可以比较顺畅, 但为什么在 hermes中就不行呢?
|这是一个非常典型的问题，*原因在于 LM Studio 的原生对话模式 与 Hermes Agent 模式 对系统资源的消耗方式完全不同。*

虽然你的 3050 8G 显卡可以流畅运行 Gemma 4 26B 的单机对话，但在 Hermes 中遇到障碍主要有以下三个深层次原因：

1.“工具提示词”占用了过多的初始空间 +

**当你直接在 LM Studio 聊天时，系统提示词可能只有几句话。 +
但在 Hermes 启动时，它会向模型发送一个极其庞大的“指令集”，**包含： +

- 28 个工具的详细定义（如何写代码、如何读文件、如何搜索等）。 +
- Agent 的行为逻辑指南。 +

这些初始内容加起来就占用了约 11,940 个 token。 +
由于你的 LM Studio 默认只给模型开了 4,096 的 Context 窗口，模型连“说明书”都没读完就“溢出”了，导致无法做出任何回应。 +

2.显存与内存的“极限拉锯” +
Gemma 4 26B 的量化版（IQ4_NL）文件大小约为 14.70 GB。 +

你的显卡 (RTX 3050)：只有 8 GB 显存。 +

现状：显卡塞不下整个模型，约有 7-8 GB 的模型数据被迫跑在速度极慢的 系统内存 (RAM) 上。 +

区别：普通对话时，模型只需要处理几十个 token 的输入；而 Hermes 每一轮对话都要带着那 1.2 万个 token 的“说明书”进行推理。在 3050 上，这种规模的计算会让推理速度从“秒回”变成“分钟级”，甚至让系统卡死。 +

3.Agent 的“思考”成本更高 +

- *普通聊天：模型只需根据你的话接下一句。* +
- *Hermes 聊天：模型在回答前需要先“思考”是否需要调用工具。它会反复阅读那 28 个工具的描述，这个预处理过程（Prompt Ingestion）在长文本下对显卡的压力呈几何倍数增长。* +
